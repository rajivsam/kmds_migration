# KMDS Toolkit Usage Summary

## Purpose

This document summarizes how the Olist migration project uses the KMDS toolchain end-to-end.
It covers the core workflow components, the actual execution path taken in this repository, and the KMDS knowledge graph artifact produced from generated documentation.

## Toolkit components and responsibilities

1. `dd-parser-cleaner`

   - Used by the data scientist to drive metadata-led profiling, entity extraction, and dataset cleaning.
   - Executed as `classify-entities` and `clean-dataset` to validate the customer/order schema and produce cleaned dataset outputs.
   - In this repo, the customer data workflow is isolated under `customer_config.yaml` and generates cleaned output at `data/dd_cleaner/olist_customers_dataset_clean.csv`.
2. Featurizer pipeline (`featurization_scripts/featurization.py`)

   - Centralizes raw data source paths, stage definitions, and output artifacts in `featurizer_config.yaml`.
   - Builds a prepared SP 2017 dataset and a weekly product revenue matrix for unsupervised analysis.
   - Writes core output artifacts such as `data/olist_daily_orders_prepared.csv`, `data/SP_2017_orders_filtered_prepared.csv`, `data/SP_2017_weekly_revenue_prepared.csv`, and `data/SP_2017_freq_prod_weekly_sales_prepared.csv`.
3. Feature advisor

   - Operates as a rule-based recommendation layer for feature engineering choices.
   - In this project, it is used in a graph-modeling intent to recommend encoding strategies for raw and derived fields.
   - Its output is recorded under `data/data/feature_advisor/feature_advisor_recommendations.md` and referenced by the featurization documentation.
4. Model advisor

   - Provides guidance on model structure, algorithm selection, and clustering configuration.
   - In the Olist workflow, the model advisor supports the spectral coclustering setup and sanity checks for the final unsupervised pipeline.
   - This component is captured in notebook documentation and modeling artifacts generated by the spectral clustering exercise.
5. Modeling implementation

   - The modeling layer executes the actual unsupervised algorithm.
   - The implementation follows the clustering modeling guidelines from the `kmds-modeling` package, documented here: [kmds-modeling modeling contracts](https://github.com/rajivsam/kmds-modeling/tree/main/documents/modeling_contracts)
   - The actual design input for this project is the detailed rationale document [Rationale for Spectral Clustering](rationale_for_spectral_clustering.md), rather than the example code alone.
   - In this repository, the pipeline is a TF-IDF normalized spectral coclustering of the SP 2017 product-week matrix.
   - The output is represented by the KMDS artifact generation flow and supporting notebooks.
6. `kmds-data-helper`

   - Acts as the bridge between generated repository artifacts and the KMDS RDF knowledge graph.
   - In this run, it ingested `output/full_service_report.json` and produced the KMDS project graph at `data/kmds/project_knowledge_graph.xml`.
   - The helper package is the key integration point for converting repository text, notebook summaries, and metadata into KMDS observations.
7. `kmds-ui`

   - The visualization layer for interactive KMDS inspection and exploration.
   - Although not executed in this run, `kmds-ui` is the natural consumer of the generated `project_knowledge_graph.xml` artifact.
   - It is the intended tool for browsing KMDS workflows, observations, and project entities after ingestion.

## Actual execution path in this repository

### Step 1: Start with dd-parser-cleaner profiling and cleaning

- The data scientist ran `dd-parser-cleaner` to classify entities and clean the customer/order datasets.
- This step established the metadata hygiene foundation and produced `data/dd_cleaner/olist_customers_dataset_clean.csv`.
- After that handoff, I collaborated with the data scientist to continue with featurization, modeling, and KMDS graph generation.

### Step 2: Generate the featurization and migration artifacts

- `featurizer_config.yaml` defines the dataset sources and feature-building stages.
- The featurization script and notebooks read raw Olist CSV files from `data/` and write prepared outputs back to `data/`.
- This includes the São Paulo 2017 weekly revenue matrix used for unsupervised analysis.

### Step 3: Run the feature advisor

- The feature advisor is invoked using project metadata and a graph-modeling intent.
- It generates recommendations for categorical encoding, numeric handling, and field review.
- The advisor output is captured in `data/data/feature_advisor/feature_advisor_recommendations.md`.

### Step 3: Execute the modeling workflow

- The spectral coclustering notebook and modeling wrapper perform the unsupervised clustering.
- The model advisor helps choose the graph-based clustering approach.
- Final artifacts from this stage include notebook outputs and model summary observations.
- The analysis found a dominant week cluster of 21 weeks, a second significant cluster of 11 weeks, and several smaller seasonal week clusters.
- Product cluster sizes were highly imbalanced, with a baseline cluster of about 8,661 products, a mid-sized cluster of 420 products, another cluster of 150 products, and many very small specialized clusters.
- These results emphasize a strong baseline demand regime plus distinct temporal and product-specific seasonal patterns, supporting both steady-state inventory planning and cluster-triggered surge planning.

### Step 4: Build the KMDS helper output

- A helper JSON artifact was created at `output/full_service_report.json`.
- This file aggregates generated documentation from `documents/`, `agent_documents/`, and notebook markdown cells into a single ingestion-ready summary.
- It is the input required by `kmds-data-helper` to build the KMDS graph.

### Step 5: Generate the KMDS knowledge graph

- The command used was:

```bash
.venv/bin/kmds-kb --workspace . \
  --project-file data/kmds/project_knowledge_graph.xml \
  --workflow-name olist_migration_kmds \
  --mode auto \
  --workflow-type application
```

- The result is the RDF graph file `data/kmds/project_knowledge_graph.xml`.
- The ingestion mode selected was `summary`, producing 422 exploratory observations from the helper output.

## What this graph represents

- The KMDS artifact is not a direct vector model output; it is an entity- and observation-level representation of the migrated workflow.
- It captures the project as a KMDS workflow, summarizes documentation and notebook evidence, and logs the exploratory model observations produced by the helper.
- This graph is now available for downstream KMDS query, inspection, or visualization.

## Notes on toolkit alignment

- `dd-parser-cleaner` provides the dataset hygiene and metadata foundation.
- The featurizer pipeline converts raw Olist data into graph-friendly prepared inputs.
- The feature advisor creates a documented rationale for encoding decisions.
- The model advisor connects the clustering choice to the unsupervised workflow.
- `kmds-data-helper` transforms project artifacts into the KMDS ontology.
- `kmds-ui` is the interface designed to expose the final graph to users and analysts.

## Summary

The KMDS toolchain in this project is a layered pipeline:

- `dd-parser-cleaner` for data quality,
- a featurizer for prepared representations,
- advisors for engineering and modeling guidance,
- a modeling implementation for clustering,
- and `kmds-data-helper` plus `kmds-ui` for knowledge graph construction and visualization.

This repository now contains both the operational dataset outputs and the converted KMDS knowledge graph artifact required to make the workflow discoverable and queryable.
